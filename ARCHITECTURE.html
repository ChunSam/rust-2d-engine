skeleton-engine 아키텍처 구조도

이 문서는 공개 API 레퍼런스가 아니라 유지보수자가 엔진 내부 실행 흐름을 빠르게 파악하기 위한 구조 문서다. API 사용법과 예제는 REFERENCE.html을 기준으로 본다.

Rust 2D engine custom ECS winit + wgpu native + WASM

전체 구조의 중심은 App과 World다. 사용자 코드는 App::new()로 기본 리소스가 들어간 World를 얻고, 엔티티/컴포넌트/리소스/시스템/씬을 등록한 뒤 App::run()으로 winit 이벤트 루프에 진입한다. 이후 매 프레임은 입력 수집, 시스템 실행, 엔진 후처리, 렌더링 순서로 진행된다.

사용자 코드에서 GPU 제출까지

src/lib.rs는 이 구조의 공개 입구다. 내부 모듈을 선언하고 App, World, System, 주요 컴포넌트와 리소스를 engine::*로 재수출한다.

프레임 실행 흐름

프레임의 실제 중심은 ApplicationHandler 구현이다. about_to_wait()가 매 루프마다 request_redraw()를 요청하고, WindowEvent::RedrawRequested에서 dt 계산, update(), 종료/리사이즈 처리, render() 호출이 이어진다.

RedrawRequested 기준 프레임 순서

1. 이벤트 수집keyboard, mouse, touch, IME는 InputState/TouchState에 기록된다.
2. dt 계산last_frame 기준으로 초 단위 delta를 만들고 0.1초로 clamp한다.
3. App::update시스템 실행과 엔진 후처리가 일어나는 메인 단계다.
4. App::renderWorld의 렌더 데이터를 읽어 wgpu command encoder에 패스를 쌓는다.
update() 내부 상세
change tracking clearWorld::clear_change_tracking() 후 현재 프레임 변경 감지를 새로 시작한다.
viewport resourcesGPU physical size와 window scale factor로 logical viewport를 갱신한다.
egui beginDebugUi가 있으면 raw input을 받아 egui pass를 시작한다.
schedule computeSystemConfig before/after를 DAG 정렬하고 cycle 시 삽입 순서로 폴백한다.
systems runSystem::run(&mut World, dt)를 exec order로 실행하고 set 비활성화와 panic 격리를 적용한다.
hierarchy + camera사용자 시스템 뒤, 렌더 직전에 계층 변환 전파와 카메라 follow/effect를 갱신한다.
debug/editor네이티브 Inspector, scene graph, gizmo, profiler UI 데이터를 준비한다.
egui endegui shapes를 tessellate하고 결과를 render()에서 사용할 임시 버퍼에 저장한다.
events flushregister_event로 등록된 Events<E> 큐를 프레임 끝에 비운다.
input flushjust pressed/released, text chars, scroll 등 프레임성 입력을 정리한다.
scene commandSceneChange에 요청된 Replace/Push/Pop을 적용한다.
asset polling핫 리로드와 async image completion을 poll하고 GPU texture를 재업로드한다.
사용자 시스템은 이벤트와 입력이 flush되기 전에 실행된다. 같은 프레임에 입력을 읽거나 이벤트를 보내고, 후속 시스템이 읽는 구조가 가능하다. 씬 전환은 시스템 실행 뒤에 처리되므로 전환 요청 시스템이 현재 프레임 실행을 즉시 중단하지 않는다.

World 데이터 모델

World는 archetype 기반 밀집 컬럼 스토리지다. 엔티티는 Entity(u32) ID이고, 컴포넌트 조합이 바뀔 때 엔티티가 다른 archetype으로 이동한다. 리소스는 TypeId 기반 싱글턴 맵으로 보관된다.

World 내부 소유 데이터

Entity lifecycle next_id, free_ids, entities, entity_location으로 생존 엔티티와 row 위치를 추적한다.
Archetype storage type_set이 같은 엔티티를 묶고 columns: HashMap<TypeId, Vec<ComponentBox>>에 컴포넌트를 저장한다.
Resources InputState, Camera, AssetServer, SceneChange, queues 같은 전역 상태를 타입별로 저장한다.
Registries Reflect registry는 Inspector 필드 편집에, clone registry는 clone_entity류 기능에 사용된다.

쿼리 이터레이터가 살아있는 동안 같은 World에서 mutable access를 다시 얻으면 borrow 충돌이 난다. 그래서 엔진 문서의 표준 패턴은 엔티티 목록을 먼저 collect()하고, 두 번째 루프에서 get_mut()로 수정하는 방식이다. 

let entities: Vec<Entity> = world.query::<Player>().map(|(e, _)| e).collect();
for entity in entities {
    if let Some(transform) = world.get_mut::<Transform>(entity) {
        transform.position.x += 10.0;
    }
}

렌더 파이프라인

렌더링은 render()가 World를 읽고 여러 렌더 큐를 drain하면서 구성한다. 애니메이션, UI, 디버그 드로우 같은 서브시스템은 렌더러를 직접 호출하지 않고 컴포넌트나 queue resource에 데이터를 남긴다.

네이티브/WASM 분기

크레이트는 cdylib와 rlib를 함께 빌드하고, 많은 기능이 cfg(target_arch = "wasm32")로 갈린다. 공개 API도 일부 re-export가 네이티브 전용이다.

플랫폼별 차이

Native rapier2d physics, rodio audio, gilrs gamepad polling, file watching, tungstenite networking, lighting/fade/GPU particle modules.
WASM wasm_bindgen start hook, canvas lookup by game-canvas, async GPU init via spawn_local, WebGL feature path, optional font data for text.

WASM에서는 GPU 초기화가 async라서 PENDING_GPU thread-local에 결과를 넣고 about_to_wait() 또는 RedrawRequested에서 finish_init()을 호출한다. 네이티브에서는 pollster::block_on(GpuContext::new(...))로 resumed() 안에서 바로 초기화한다.

확장 지점

  • 새 시스템
    System을 구현하고 app.add_system 또는 add_system_labeled로 등록한다. 씬 전용이면 Scene::on_enter에서 시스템 벡터에 추가한다.
  • 시스템 순서
    SystemConfig::label, before, after, in_set을 사용한다. cycle이 있으면 로그를 남기고 삽입 순서로 폴백한다.
  • 새 컴포넌트
    해당 모듈에 타입을 정의하고 필요 시 src/lib.rs에서 re-export한다. Inspector 편집이 필요하면 reflect 등록 흐름을 맞춘다.
  • 새 리소스
    world.insert_resource로 넣고 시스템에서 resource/resource_mut로 접근한다. 기본 리소스라면 App::new와 reload_scene 양쪽을 확인한다.
  • 새 이벤트
    app.register_event::<E>()로 Events<E> 리소스와 프레임 flush hook을 등록한다.
  • 씬 전환
    시스템에서 SceneChange::request(SceneCmd::...)를 호출한다. 실제 적용은 시스템/입력/event flush 뒤에 일어난다.
  • 렌더 데이터
    렌더러를 직접 호출하기보다 컴포넌트나 UiQueue, UiImageQueue, TextQueue, DebugDraw 같은 리소스에 그릴 데이터를 남긴다.
  • 에셋
    App::load_image, load_image_async, load_atlas, load_script를 통해 AssetServer와 GPU 업로드 경로를 함께 태운다.

유지보수 체크리스트

  • App::new에 기본 리소스를 추가하면 reload_scene에도 같은 초기화를 반영한다.
  • 새 공개 API는 src/lib.rs re-export와 REFERENCE.html 문서 필요 여부를 함께 확인한다.
  • 렌더 레이어를 넘나드는 변경은 시스템이 데이터를 준비하고 렌더러가 단순히 읽는 구조를 유지한다.
  • 입력/이벤트/씬 전환 타이밍 변경은 프레임 흐름 전체에 영향이 있으므로 예제와 UI 시스템을 같이 확인한다.
  • 네이티브 전용 기능을 추가하면 cfg(not(target_arch = "wasm32")) 공개 re-export와 WASM 빌드를 확인한다.
  • 문서 변경 뒤에는 최소한 cargo package --locked --list로 패키지 포함 목록을 확인한다.
